# Lenso Service Product Layer V6 Design

## Goal

Make services a complete Lenso product surface. V5 separated service and module
language; V6 should make the separation useful in daily work:

- service authors can create Rust or TypeScript services quickly;
- host developers can install, run, check, and uninstall services predictably;
- operators can see service state, health, calls, and story links in Console;
- catalog entries can map business modules to provider services;
- examples prove that one service can provide multiple modules.

This is still not a service mesh, API gateway, orchestration layer, distributed
transaction system, or schema registry.

## Baseline

The current service boundary is:

```text
service = remote process + service manifest + one or more provided modules
module = business capability inside a service or linked into the host
host = control plane for auth, capabilities, runtime queues, outbox, story,
       technical operations, service source config, and install state
```

V6 keeps that boundary. The product change is that the service layer becomes
first-class across contract, SDKs, CLI, Console, catalog, and examples.

## Non-Goals

V6 must not add:

- service discovery;
- API gateway routing;
- service mesh concepts;
- distributed transactions;
- schema registry;
- automatic package trust;
- Kubernetes or Compose orchestration beyond exportable local/deployment hints;
- new host-owned runtime abstractions unless the existing protocol cannot carry
  the V6 behavior.

## Approach

Use one shared service contract and make every surface consume it.

```text
lenso.service.json
  -> Rust service SDK
  -> TypeScript service SDK
  -> lenso service create/dev/check/install
  -> Runtime Console Service Center
  -> catalog provider mapping
  -> examples and contract tests
```

This avoids duplicating service truth across templates, docs, and Console.

## Service Contract V1

`lenso.service.json` becomes the service product contract. It should support:

- provider identity: service name, version, vendor, summary, homepage;
- protocol metadata: manifest version, minimum host version, SDK language and
  SDK version when generated by an SDK;
- provided modules: module names, versions, capabilities, dependencies, routes,
  admin surfaces, runtime functions, event handlers, and console packages;
- config schema: host-visible config keys, defaults, required values, and
  secret hints;
- env schema: required environment variables, optional variables, and example
  values;
- health checks: manifest URL, ready URL, liveness URL, optional diagnostics
  URL;
- local process hints: command, args, cwd, env, auto start, ready timeout;
- compatibility flags: protocol version, supported transport features, and
  declared optional features.

The manifest remains JSON so SDKs in later languages can generate it without a
Rust dependency.

## Rust Service SDK

Add a Rust service authoring path for teams that want type safety and the same
language as the host:

```sh
lenso service create support-ticket-provider --lang rust
```

The generated service should include:

- manifest builder;
- module declaration helpers;
- HTTP route, admin action, runtime function, and event handler registration;
- health and ready routes;
- Host request header validation helpers;
- a minimal contract check test that starts the service and validates the
  manifest and ready endpoint.

Do not add a framework abstraction with one implementation. Use Axum in the
template and expose small helpers for the Lenso protocol pieces.

## TypeScript Service SDK

Add a TypeScript service authoring path for fast third-party and frontend-team
providers:

```sh
lenso service create support-ticket-provider --lang ts
```

The generated service should include:

- typed manifest builder;
- route, action, runtime function, and event handler registration;
- health and ready endpoints;
- a local `lenso service check` script;
- package metadata ready for later publishing.

`@lenso/service-kit` is the natural home for the shared TS helpers. Keep the SDK
thin and avoid reimplementing a full HTTP framework.

## CLI Workflow

V6 CLI should make the service lifecycle obvious:

```sh
lenso service create <name> --lang rust
lenso service create <name> --lang ts
lenso service dev
lenso service check <service>
lenso service install <manifest>
lenso service uninstall <service>
lenso service status
lenso service doctor
```

Behavior:

- `create` scaffolds a standalone service package, not a module;
- `dev` starts declared local services and the host, watches service readiness,
  and streams simple logs;
- `check` validates manifest, health, declared routes, actions, runtime
  functions, and event handlers where a safe probe exists;
- `install` writes `REMOTE_MODULES`, service process state, install receipts,
  env hints, and Console extension state;
- `uninstall` removes service source state, service process state, install
  receipts, and Console extension state;
- `module install <business-module>` resolves business modules from catalog and
  may install the provider service as an implementation detail.

## Runtime Console Service Center

Add a Console surface focused on services, not modules:

- service list with installed, configured, ready, unhealthy, restart pending,
  and incompatible states;
- provider detail page with manifest, config, env hints, health checks, and
  provided modules;
- service-to-module graph;
- recent remote calls grouped by service and module;
- Runtime Story links that identify the service operation behind each node;
- Technical Operations links for retries, failures, and admin actions;
- local managed service actions: restart, copy command, copy env hints, and view
  recent logs when the host exposes them.

Console should read host-owned service state. It should not invent a separate
source of truth.

## Catalog And Install Model

Catalog stays curated, but entries can now distinguish business modules from
provider services:

```json
{
  "module": "support-ticket",
  "providedBy": "support-suite-provider",
  "serviceManifest": "https://example.com/lenso/service/v1/manifest",
  "summary": "Ticket intake, triage, and operations"
}
```

Rules:

- `lenso module install support-ticket` installs the business module path;
- the catalog may install `support-suite-provider` if the module is service
  provided;
- `lenso service install <manifest>` installs a provider directly;
- uninstalling a service removes all service-owned local state, but must warn
  when installed business modules still depend on it.

## Security And Policy

The host remains the control plane:

- browser bearer tokens do not pass through to services;
- Host-to-Service calls use host-issued service call metadata;
- capability checks happen before the Host invokes a service operation;
- services can perform defense-in-depth checks on module-local data;
- services cannot write host runtime, outbox, story, or technical operation
  tables directly;
- env values and secrets are written only to host-local config files or operator
  instructions, never to catalog JSON.

## Business Proof

Use `support-ticket` as the V6 proof because it is real enough to exercise the
service model:

```text
support-suite-provider service
  provides support-ticket module
  provides support-notification module
  provides support-knowledge-base module
```

The proof should show:

- one service with multiple modules;
- one module installed from catalog;
- provider service installed as part of that module install;
- HTTP route, admin action, runtime function, and event handler all visible in
  Console;
- Remote Calls and Runtime Story linking back to the service operation;
- `lenso service check` failing clearly when the service is stopped.

## Phases

### Phase 1: Contract And Templates

- finalize `lenso.service.json` fields for V6;
- add Rust and TS service templates;
- add minimal SDK helpers where templates would otherwise duplicate protocol
  boilerplate;
- add contract checks that templates can run locally.

### Phase 2: CLI Dev And Check

- add `lenso service create --lang rust|ts`;
- add `lenso service dev`;
- strengthen `lenso service check` and `doctor`;
- align old remote wording to service wording.

### Phase 3: Console Service Center

- add service list and detail state;
- wire service health, provided modules, remote calls, Runtime Story, and
  Technical Operations;
- keep actions small: restart managed local service, copy command, copy env.

### Phase 4: Catalog Provider Mapping

- update catalog DTOs and fixtures for `providedBy` service mappings;
- make module install resolve provider services;
- make uninstall warn about dependent modules.

### Phase 5: Support Suite Proof

- convert support-ticket into the main V6 service proof;
- add sibling support modules inside the same provider service;
- document the run from create to install to Console observation.

## Verification

Focused checks are enough for V6:

- Rust service template contract check;
- TS service template contract check;
- CLI `service create`, `service check`, `service install`, and `service
  uninstall` fixture tests;
- Console model tests for service states and provider-module graph;
- support-suite example runbook check;
- regression check that linked modules still load and existing service protocol
  manifests still work.

Avoid broad smoke loops unless the change touches the actual end-to-end
service-host-Console path.

## Open Decisions

- Rust SDK crate name: prefer `lenso-service` unless crate naming conflicts with
  existing package strategy.
- TypeScript SDK package name: keep `@lenso/service-kit` unless publishing
  policy requires a split.
- `service dev` scope: start with local process hints and host startup only;
  deployment orchestration stays outside V6.
- logs in Console: show recent host-captured managed-service logs only when
  available; do not build a logging backend.

## Success Criteria

V6 is done when a developer can:

1. create a Rust or TypeScript service;
2. run it locally with the host;
3. install it through a service manifest or business module catalog entry;
4. see service readiness and provided modules in Console;
5. invoke route, action, runtime function, and event handler paths;
6. diagnose stopped, incompatible, or partially installed services with one CLI
   command;
7. understand why services and modules are different without reading internal
   architecture code.
